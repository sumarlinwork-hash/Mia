### 1. Payment Gateway & Payout System

**Current Implementation (Basic)**:
- Simulated payment processing for development/testing
- Invoice lifecycle: create → pending → paid
- Transaction recording with gateway transaction IDs
- Creator payout queue with status tracking (pending → processed)

**API Endpoints**:
```
POST /api/discovery/payment/gateway
  Body: { invoice_id, payment_method, card_token }
  Response: { status, transaction_id, invoice }
  
POST /api/discovery/payment/payout
  Body: { creator_id, amount, payout_method }
  Response: { status, payout_id, payout }
  
POST /api/discovery/payment/payout/{payout_id}/process
  Response: { status, message }
```

### 2. Security Scanner & Signature Verification

**Current Implementation (Basic)**:
- Static code analysis for 6 suspicious patterns:
  - `os.system()`, `subprocess.call()`
  - `exec()`, `eval()`, `__import__()`, `compile()`
- SHA256 hash computation for file integrity
- Severity classification: low, medium, high
- Code size analysis (>50KB flagged)

**API Endpoints**:
```
POST /api/discovery/security/scan
  Body: { code }
  Response: { clean, findings: [{ pattern, severity, description }], severity, hash }
  
POST /api/discovery/security/verify
  Body: { file_path, expected_hash }
  Response: { status, matches, actual_hash, expected_hash }
```

**Future Enhancements**:
- Integrate ClamAV or similar antivirus engine
- Dynamic sandbox execution monitoring
- Behavior analysis during test execution
- Dependency vulnerability scanning (CVE database)
- Code signing certificates (PKI)
- Automated quarantine for malicious items
- Security audit trail in marketplace UI

**Security Policy**:
- Items with HIGH severity findings: auto-reject
- Items with MEDIUM severity: manual review required
- Items with LOW severity: warning displayed to users
- Verified publishers can bypass some checks (trusted status)

### 3. Personalization Engine

**Current Implementation (Basic)**:
- User behavior tracking: installed, executed, searched, viewed
- Collaborative filtering: find users with similar installs
- Category matching: recommend items in user's preferred categories
- Trending score boost based on global popularity
- Cold start handling: return trending items for new users

**API Endpoints**:
```
POST /api/discovery/personalization/track
  Body: { user_id, action, item_id }
  Response: { status }
  
GET /api/discovery/personalization/recommendations?user_id=xxx&limit=20
  Response: [ items... ]
  
GET /api/discovery/personalization/profile/{user_id}
  Response: { user_id, exists, installed_count, executed_count, ... }
```

**Scoring Algorithm**:
```
score = 0

# Collaborative filtering (similar users)
for each other_user with overlapping installs:
  if item installed by other_user:
    score += 5.0

# Trending boost
score += min(trending_score * 0.5, 10.0)

# Category matching
if item.category in user.preferred_categories:
  score += 8.0

return items sorted by score DESC
```

**Future Enhancements**:
- ML-based recommendation (Matrix Factorization, Neural Collaborative Filtering)
- Real-time behavior tracking with event streaming
- A/B testing framework for recommendation algorithms
- Contextual recommendations (time of day, current task)
- Diversity injection to avoid filter bubbles
- Exploration/exploitation balance
- User feedback loop (like/dislike recommendations)

**Data Models**:
```json
{
  "user_history": {
    "user123": {
      "installed": [
        { "item_id": "story_weaver", "timestamp": "...", "metadata": {} }
      ],
      "executed": [],
      "searched": [],
      "viewed": []
    }
  },
  "item_trending": {
    "story_weaver": 15,
    "media_curator": 12
  }
}
```

### 4. Recommendation Feed

**Current Implementation (Basic)**:
- Trending items endpoint with composite scoring:
  - Trending score (40%): recent user interactions
  - Downloads (30%): historical popularity
  - Rating (30%): quality metric
- Personalized recommendations per user (see Personalization Engine)
- Limit parameter for pagination

**API Endpoints**:
```
GET /api/discovery/personalization/trending?limit=20
  Response: [ items... ]
  
Composite Score = (trending * 2.0) + (downloads / 1000.0) + (rating * 2.0)
```

**Future Enhancements**:
- "For You" feed with ML-powered ranking
- "Trending Now" with time-decay (last 24h, 7d)
- "New & Noteworthy" for recently published items
- "Similar to {item}" recommendations
- Social proof ("100 users in your industry installed this")
- Seasonal/trending topics integration
- Sponsored items support (monetization)

**Feed Types**:
1. **Personalized**: Based on user history and preferences
2. **Trending**: Global popularity across all users
3. **Editorial**: Curated by marketplace team (featured items)
4. **Similar**: Content-based filtering by category/tags
5. **New**: Recently published items (last 30 days)

### 5. Full Sandbox Isolation (NOT YET IMPLEMENTED)

**Planned Implementation**:
- Docker container per skill execution
- Resource limits:
  - CPU: 0.5-2 cores max
  - Memory: 256MB-1GB max
  - Network: optional (configurable per skill)
  - Disk: 100MB temp storage
- Isolated filesystem:
  - Read-only access to skill code
  - Write access only to /tmp
  - No access to host filesystem
- Environment variables:
  - No access to host secrets
  - Injected skill-specific config only
- Timeout enforcement: 30s max execution

**Architecture**:
```
User Request → API → Docker API → Container Spawn
  → Load Skill → Execute → Capture Output
  → Container Destroy → Return Result
```

**Security Boundaries**:
- No network access by default (allowlist per skill)
- No subprocess spawning
- No raw socket access
- No privileged operations
- Automated cleanup on timeout/error

**Implementation Options**:
1. **Docker SDK for Python**: Direct container management
2. **Firecracker microVMs**: Stronger isolation, faster startup
3. **gVisor**: Application kernel for sandboxing
4. **WebAssembly (WASM)**: Lightweight sandboxing (future)

**Estimated Effort**: 2-3 sprints
**Priority**: Medium (important for enterprise, less critical for trusted publishers)
